# Cookie,Session,Token 的区别

# http 无状态

用户访问服务器后，关闭网页，结束会话，下次再次访问时服务器并不知道又是你。就是说这一次请求和上一次请求是没有任何关系的，互不认识的，没有关联的。这种无状态的的好处是快速。

# cookie 和 session

由于 http 的无状态性，为了使某个域名下的所有网页能够共享某些数据 (也就是不用用户频繁的输入用户名和密码)，session 和 cookie 出现了。客户端访问服务器的流程如下：

  • 客户端发送一个 http 请求到服务端
  • 服务端接收请求后，建立一个 session，sessionId 存在数据库，并发送一个 http 响应到客户端，这个响应头就包含 Set-Cookie 头部，该头部包含了 sessionId。Set-Cookie 的格式如下：
  • 在客户端发起的第二次请求，如果本地有服务器给的 Set-Cookie，并且这个 cookie 没有过期失效，浏览器会自动在请求头中添加 cookie
  • 服务器接收请求，分解 cookie，验证信息，核对成功后返回 response 给客户端

# token

随着互联网的发展，用户群体变得很大，如果服务器依旧使用基于 Cookie 的 Session, 在特定时间有大量用户访问服务器的时候，服务器需要存储大量的 sessionId，如果有多台服务器，一台服务器存储了 sessionId 需要面临分享 sessionId 给其他服务器，涉及到分布式 session 的问题。在各种原因和需求的情况下，出现了一个技术叫 JWT, 全称 JSON Web Token。

  • 首先用户登录网页
  • 服务器会生成一个 JWT，服务器不需要保存 JWT，只保存 JWT 签名的密文，然后把 JWT 返回给客户端浏览器
  • 浏览器可以以 Cookie 或 Storage 的形式存储下来，注意是存在客户端。
  • 客户端在每次发送请求都会把 JWT 也发送给服务器，用户就不需要重新输入账号密码，和 session 类似，区别在于 token 存储在用户端

# JWT

JWT 由三部分组成，由。分开：

  • header: 声明用什么算法来生成签名
  • payload: 特定的数据，比如有效期
  • signature：签名信息。header 和 payload 两部分的内容会经由 Base64 编码生成两段编码，这两段编码经过 header 中声明的算法运算生成最终的签名信息，签名信息就是 signature 部分

这样一个完整的 JWT 就可以发送给客户端了。

# 三者的区别

  • session 是诞生并保存在服务器那边的，由服务器主导一切，Cookie 则是一种数据载体，把 session 放入 cookie 中送到客户端那边，cookie 跟随 http 的每个请求发送出去；
  • token 是诞生在服务器，但保存在浏览器这边，由客户端主导一切，可以放在 Cookie 和 Storage 里，持有 token 就像持有 "令牌" 一样可以允许访问服务器。
  • token 可以抵抗 CSRF 攻击，cookie+session 不行
  • token 不存在负载均衡问题，在服务器采用分布式和集群的环境时，session 面临负载均衡问题，需要采取分布式 session。
