# 面试题

# 1、spring cloud 的五大组件是什么？

  • 服务注册发现：Eureka/Zookeeper 注册中心 (尤里卡)
  • 客户端负载均衡：Ribbon/Feign (负载均衡)
  • 断路器 / 熔断器：Hystrix (海斯特里克斯)
  • 服务网关：Zuul (祖尔)
  • 分布式配置中心：Config

# 2、Eureka 是怎么注册服务的？

Eureka 包含两个组件：Eureka Server 和 Eureka Client。

  • Eureka Server 提供服务注册服务，各个节点启动后，会在 Eureka Server 中进行注册，这样 EurekaServer 中的服务注册表中将会存储所有可用服务节点的信息，服务节点的信息可以在界面中直观的看到。
  • Eureka Client 是一个 java 客户端，用于简化与 Eureka Server 的交互，客户端同时也就是一个内置的、使用轮询 (round-robin) 负载算法的负载均衡器。

# 3、重写与重载的区别？

  • 重写 (Override)： 子类把父类中的方法重新实现一遍。

    1. 方法名，参数，返回类型都跟父类一样。
    2. 访问权限要大于父类方法。
    3. 不能抛出比父类更宽泛的异常（里氏代换原则）

  • 重载 (Overload)： 方法名相同，参数列表不同，返回类型也可以不同（也可以相同）

    1. java 多态性的表现
    2. 参数不同（包括类型，个数，顺序）
    3. 返回类型可以相同也可以不同。

  • 面试时，问：重载（Overload）和重写（Override）的区别？
    答：方法的重载和重写都是实现多态的方式，区别在于前者实现的是编译时的多态性，而后者实现的是运行时的多态性。重载发生在一个类中，同名的方法如果有不同的参数列表（参数类型不同、参数个数不同或者二者都不同）则视为重载；重写发生在子类与父类之间，重写要求子类被重写方法与父类被重写方法有相同的参数列表，有兼容的返回类型，比父类被重写方法更好访问，不能比父类被重写方法声明更多的异常（里氏代换原则）。重载对返回类型没有特殊的要求，不能根据返回类型进行区分。

# 4、StringBuffer 和 StringBuilder 区别

都是可变字符序列

  1. StringBuffer 线程安全，StringBuilder 线程不安全；
  2. StringBuilder 性能更好，没有加锁同步操作，一般开发中用的多；
  3. 在对安全性要求高的开发中，比如医院，银行，用 StringBuffer, 追求效率的系统中，比如新闻网站，论坛用 StringBuilder。

# 5、mysql 的索引以及 sql 优化

  • 简单的说就是：索引指数据库的目录

  • 定义：索引是关系型数据库中给数据库表中一列或者多列的值排序后的储存结构，SQL 的主流索引结构有 B + 树以及 Hash 结构

  • 索引种类： 普通索引，主键索引，唯一索引，组合索引

  • 在什么情况下加索引

    1. 主键自动建立索引
    2. 频繁查询的字段应该加索引
    3. 外键关系加索引
    4. 查询中排序的字段
    5. 查询统计或者分组的字段

  • 怎么看用没用索引：通过 explain 查询 sql 执行计划，主要看 key 使用的是哪个索引

  • sql 优化怎么做

    • 一、首先开启数据库慢查询日志，定位到查询效率比较低的 sql，找出相应的 sql 语句进行分析

      1. 表设计是否规范，是否符合三范式标准
      2. 查看数据表是否冗余
      3. 查看字段类型是否合理，用 varchar 代替 char，能用数值的话不用字符存储
      4. 避免使用 null

    • 二、查看 sql 语句是否规范

      1. 避免使用关键字：or,in,not in,!=,<>, 避免使用 select*
      2. 避免子查询

    • 三、分析 sql 的索引是否可以用上

      1. explain 查询 sql 的执行计划，重点关注的几个列就是，type 是不是全表扫描
      2. 看 key 使用的是哪个索引，没加索引时，key 的值为 NULL，加了索引后的值为 address，即决定查询结果用到了索引 address
      3. 看一下 rows 扫描行数是不是很大，没加索引时，rows 的值为 17，即数据表 student 中所有数据，说明没加索引时的 SQL 查询是全表扫描；加了索引后，rows 的值为 6。

# 6、GC 的了解，堆里面的 GC 有哪些种类？

  • GC (GarbageCollection) 是垃圾回收机制（垃圾回收器），GC 是 JVM 对内存进行管理的方式，堆是垃圾收集器管理的主要区域。因为收集器采用的分代收集算法，堆被分为新生代和老年代，新生代又被分为 Eden 和两个 Survivor 区域（from 和 to），比例是 Eden:from:to=8:1:1
  • 种类：
    1. Serial GC
    2. ParNew GC
    3. CMS GC
    4. Parrallel GC
    5. G1 GC
    6. jadk11 提出的 ZGC

# 7、java 的内存结构

  • 堆，虚拟机栈，本地方法栈，方法区，程序计数器。
  • 堆和方法区是线程共享区域，其余都是线程私有；
  • 虚拟机栈存着很多栈帧，每个方法执行都会创建一个栈帧，栈帧包含局部变量表，操作数栈，动态链接，方法返回地址；
  • 方法区存放加载的类信息，常量池，静态变量。

# 8、spring Bean 的生命周期？

  • 往大了说，主要包含四个阶段，分别是：实例化、属性赋值、初始化、销毁；

  • 实例化 Bean: 当客户向容器请求一个未初始化的 Bean 是，容器会调用 doCreateBean () 方法进行实例化，通过反射创建一个 bean 对象；

  • 属性填充：Bean 实例创建出来后，就给这个 Bean 对象进行属性填充，也就是注入这个 Bean 依赖的其他 bean 对象；

  • 初始化：属性填充完后，进行初始化 Bean 操作，分为几个步骤：

    1. 执行 Aware 接口的方法：Spring 会检测该对象是否实现了 xxxAware 接口，通过 Aware 类型的接口，可以让我们拿到 Spring 容器的些资源。如实现
      BeanNameAware 接口可以获取到 BeanName，实现 BeanFactoryAware 接口可以获取到工厂对象 BeanFactory 等；
    2. 执行 BeanPostProcessor 的前置处理方法 postProcessBeforelnitialization ()，对 Bean 进行一些自定义的前置处理；
    3. 判断 Bean 是否实现了 InitializingBean 接口，如果实现了，将会执行 lnitializingBean 的 afterPropertiesSet () 初始化方法；
    4. 执行用户自定义的初始化方法，如 init-method 等；
    5. 执行 BeanPostProcessor 的后置处理方法 postProcessAfterinitialization ()

  • 销毁：初始化完成后，Bean 就成功创建了，之后就可以使用这个 Bean，当 Bean 不需要时，会进行销毁

    1. 首先判断 Bean 是否实现了 DestructionAwareBeanPostProcessor 接口，如果实现了，则会执行 DestructionAwareBeanPostProcessor 后置处理器的销毁回调方法；
    2. 其次会判断 Bean 是否实现了 DisposableBean 接口，如果实现了将会调用其实现的 destroy () 方法；
    3. 最后判断这个 Bean 是否配置了 dlestroy-method 等自定义的销毁方法，如果有的话，则会自动调用其配置的销毁方法。

  • 循环依赖问题：循环依赖其实就是循环引用，也就是两个或则两个以上的 bean 互相持有对方，最终形成闭环。一般性的循环依赖，程序会一直执行下去最终导致内存溢出。spring 三级缓存可以解决单例的 setter 方法注入循环依赖。构造器循环依赖是无法解决的。

# 9、springboot 两大核心？

  • 自动化配置：简化配置核心；基于 Configuration，EnableXX，Condition。
  • spring-boot-starter: 脚手架核心；整合各种第三方类库，协同工具。

# 10、hashmap 底层原理

  1. 数据结构：hashmap 的数据结构为数组 +（链表或红黑树），为什么采用这个结构，因为数组查询效率高，插入删除效率低，而链表查询效率低，插入删除效率高，hashmap 结合两者的结构，兼备两者的优点。数组默认大小 16

  2. 特点：快速存储，快速查找，可伸缩（列表长度超过 8 会变成红黑树）

  3. 采用的 hash 算法：java 所有的对象都有 hashcode。hashmap 的数组下标计算方式：

  • hash 值 = (hashcode)^(hashcode>>>16); 右移异或。这样做是为了计算出来的 hash 值足够随机，足够分散，还有产生的数组下标足够随机。
  • 数组下标：hash&(16-1) = hash%16；因为数组默认大小 16，所以模 16
  • hash 冲突，同样的数组下标会产生 hash 冲突，采用单线链表解决，链表长度大于 8 会变成红黑树。
  1. 数组扩容：hashmap 数组默认长度是 16，存储比例达到 75% 会触发扩容，也就是存了 160.75=12 的时候，数组长度会变成两倍，即 162=32。

# 11、线程与进程区别？

  1. 进程是资源分配的最小单位，线程是程序执行的最小单位（资源调度的最小单位）；
  2. 进程有自己的独立地址空间，每启动一个进程，系统就会为它分配地址空间，建立数据表来维护代码段、堆栈段和数据段，这种操作非常昂贵。而线程是共享进程中的数据的，使用相同的地址空间，因此 CPU 切换一个线程的花费远比进程要小很多，同时创建一个线程的开销也比进程要小很多。
  3. 线程之间的通信更方便，同一进程下的线程共享全局变量、静态变量等数据，进程的通信机制相对很复杂，譬如管道，信号，消息队列，共享内存，套接字等通信机制，而线程由于共享数据段所以通信机制很方便。不过如何处理好同步与互斥是编写多线程程序的难点。
  4. 但是多进程程序更健壮，多线程程序只要有一个线程死掉，整个进程也死掉了，而一个进程死掉并不会对另外一个进程造成影响，因为进程有自己独立的地址空间

# 12、创建线程的方式？

  1. 继承 Thread 类
  2. 实现 Runnable 接口
  3. 通过 Executor Service 和 Callable 实现有返回值的线程
  4. 基于线程池的 execute ()，创建临时线程，实际开发用的多。

# 13、进程和线程的生命周期

  • 进程：新建，就绪，运行，阻塞，结束
  • 线程：新建、就绪、运行、阻塞、死亡

# 14、接口和抽象类的区别

  • 相同点：

    1. 都不能被实例化；
    2. 都可以包含抽象方法，且子类必须实现这些抽象方法。

  • 区别：

    1. 接口只有定义，不能有方法实现，抽象类可以有定义与实现，方法可在抽象类中实现；
    2. 一个类可以实现多个接口，但只能继承一个抽象类；
    3. 抽象类中成员变量可以是各种类型的，接口中的成员变量只能是 public static final 类型的，并且必须赋值，否则无法通过编译

  • 什么时候使用

    1. 如果有一些方法，其中的一些要有默认实现，则用抽象类；
    2. 如果想实现多重继承，必须用接口
    3. 如果基本功能在不断改变，用抽象类，因为使用接口会改变所有实现该接口的类。
